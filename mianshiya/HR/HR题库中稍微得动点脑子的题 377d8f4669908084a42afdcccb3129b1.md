**站会（Daily Stand-up / Daily Scrum）**

- **每天一次**，通常早上
- **站着开**（逼大家说重点，不长篇大论）
- 每个人轮流回答三个经典问题：
    1. 昨天我完成了什么？
    2. 今天我计划做什么？
    3. 有没有遇到阻碍 / 需要帮助？
    
    总共15min
    

**回顾会（Sprint Retrospective）**

- **每个迭代（Sprint）结束后**，验收会之后，下一个迭代规划会之前
- 全员参加（含产品经理、测试）
1. 回顾这个迭代做得好的地方
2. 找出问题 / 槽点
3. 确定 1~3 个下个迭代要改进的行动项

总共30-60min

**Sprint Review（评审会）**

**展示成果、收集反馈（演示可运行的功能）**

评审会对外展示，回顾会对内检讨

———————————

**站会**：今天干啥

**评审会**：做出来啥

**回顾会**：怎么干得更好

# 第三组

![image.png](HR%E9%A2%98%E5%BA%93%E4%B8%AD%E7%A8%8D%E5%BE%AE%E5%BE%97%E5%8A%A8%E7%82%B9%E8%84%91%E5%AD%90%E7%9A%84%E9%A2%98/image%205.png)

CI/CD是什么？

| **CI** | Continuous Integration | 持续集成 | 代码合并后自动构建、测试 |
| --- | --- | --- | --- |

| **CD** | Continuous Delivery / Deployment | 持续交付 / 持续部署 | 自动发布到测试/生产环境 |
| --- | --- | --- | --- |

CI解决什么问题？

- 多人开发，各自在本地写代码；等到最后合并时，出现大量冲突、测试失败；“在我电脑上是好的”现象频发

CI怎么解决这个问题的？

- 每天多次**将代码合并到主干**
- 每次合并**自动触发**：
    1. 拉取最新代码
    2. 安装依赖
    3. 编译/构建
    4. 运行单元测试
    5. 报告结果（成功/失败）
- 典型工具：Jenkins、GitLab CI、GitHub Actions、CircleCI

CD解决什么问题?

——CI 通过后，自动构建**可发布的产物**（如 Docker 镜像、jar 包）并部署

# 第四组 技术债务

![image.png](HR%E9%A2%98%E5%BA%93%E4%B8%AD%E7%A8%8D%E5%BE%AE%E5%BE%97%E5%8A%A8%E7%82%B9%E8%84%91%E5%AD%90%E7%9A%84%E9%A2%98/image%206.png)

**技术债务（Technical Debt）：**

一、是什么？ 为了**短期快速交付**而在代码/架构中采取牺牲代码质量的“捷径”或“不完美的实现”，这些**会像债务一样在未来产生利息（额外成本）**。

> 通俗理解：你现在写得爽，以后改得想哭。
> 

二、常见技术债务：

| 代码债务 | 重复代码、函数过长、命名混乱 |
| --- | --- |
| 设计债务 | 模块严重耦合、没有分层 |
| 测试债务 | 没有单元测试、覆盖率低 |
| 文档债务 | 没有注释、没有架构文档 |
| 架构债务 | 单体无法拆分、技术栈过时 |

三、解决办法

**1. 识别债务**

- Code Review 标记“这里以后要改”
- 维护“技术债务清单”（如 Jira 中的 Debt 标签）

**2. 计划偿还**

- 每个迭代留 10%~20% 时间做重构
- 不追求“彻底还清”，而是**控制利息**

**3. 常见偿还动作**

- 重构重复代码 → 提取公共函数
- 补写关键路径的单元测试
- 拆分过大的模块
- 统一代码格式（自动格式化工具）

**4. 预防新的债务**

- 代码规范 + 自动化检查（Lint）
- Code Review 把关
- 持续重构（童子军规则：离开时比进入时更干净）

# 第五组 敏捷开发

![image.png](HR%E9%A2%98%E5%BA%93%E4%B8%AD%E7%A8%8D%E5%BE%AE%E5%BE%97%E5%8A%A8%E7%82%B9%E8%84%91%E5%AD%90%E7%9A%84%E9%A2%98/image%208.png)

敏捷开发

一、敏捷开发是什么？-**敏捷开发**是一套**软件开发理念。**

- **快速迭代**（而不是一次做完所有事）
- **响应变化**（而不是死守计划）
- **持续交付可用软件**（而不是写大量文档）
- **人与协作**（而不是流程与工具）

> 简单说：**小步快跑、频繁交付、及时调整**。
> 

二、敏捷开发和瀑布式开发的区别？

Waterfall开发：需求→设计→开发→测试→上线，严格顺序。最后一次性交付。适合需求明确、变化少的场景，如航天、基建。

敏捷Agile开发：每个迭代都包含所有阶段，每1-4周交付一个可用版本。
**螺旋式开发**：结合瀑布+原型+风险分析，主要用于大型、高风险项目（如NASA），不是“一般现代软件开发”的默认答案。

三、主流的Agile有哪些？

**Scrum： 站会、评审会、回顾会**

**Kanban（看板）：可视化任务流动（待办→进行中→完成）**

**XP（极限编程）：TDD（测试驱动开发）、结对编程、持续集成、重构**

**PDCA 循环（戴明环）**

| **阶段** | **英文** | **核心动作** |
| --- | --- | --- |
| P | Plan（计划） | 确定目标、分析现状、制定改进方案 |
| D | Do（执行） | 按计划执行（小规模试点） |
| C | Check（检查） | 对照目标，检查执行结果，分析差距 |
| A | Act（处理/改进） | 标准化成功经验，或进入下一轮 PDCA 继续改进 |

**鱼骨图分析法（石川图 / 因果图）**

**寻找问题根本原因**的结构化工具，把可能的“原因分支”按类别画出来，形状像鱼骨。

  原因类别       原因类别

# 第九组 高并发处理

![image.png](HR%E9%A2%98%E5%BA%93%E4%B8%AD%E7%A8%8D%E5%BE%AE%E5%BE%97%E5%8A%A8%E7%82%B9%E8%84%91%E5%AD%90%E7%9A%84%E9%A2%98/image%2015.png)

**同步阻塞式VS异步非阻塞式**

一、同步阻塞式：一个请求（比如一个用户下单）在处理过程中，会**一直占用当前线程**，并且线程要**等待**数据库、下游服务或磁盘 I/O 完成之后，才能继续做下一步，最终返回结果。

**二、异步非阻塞方式**（比如 Spring WebFlux、Netty）：

- 线程发起 I/O 后不等待，立即去处理其他请求。
- I/O 完成后通过回调或事件通知继续处理。
- 同样硬件下可以支撑高得多的并发。

三、消息队列异步处理算是异步非阻塞方式么？

| **消息队列 + 立即返回** | ❌ 不阻塞（快速返回） | ✅ **架构/业务层面的异步非阻塞** |
| --- | --- | --- |
| Netty / WebFlux（真正的 NIO） | ❌ 不阻塞 | ✅ **内核/线程模型层面的异步非阻塞** |

为什么？消息队列异步处理，例子：用户请求进来后，系统**只把订单信息写入消息队列（比如 RocketMQ、Kafka）**，立刻返回“订单已接收，处理中”。后端消费者再从队列里取任务，慢慢处理库存、支付等逻辑。

所以只是对消费者而言，没有阻塞，而已
